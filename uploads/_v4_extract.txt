1. Introduction


1.1  Background

The Malawi AIDS Counselling and Resource Organisation (MACRO) currently manages its project activities, procurement processes, and financial transactions using paper-based forms. These include Vehicle/Equipment Maintenance and Service Request Forms, Payment Request Forms, Payment Vouchers, Advance Request Forms, Scope of Work documents, Attendance Sheets, Expense Reports, Goods Received Notes, Local Purchase Orders, and Advance Liquidation Checklists, among others.

While this paper-based approach has served the organisation, it presents significant challenges in terms of traceability, retrieval, approval timeliness, and accountability to donors such as PEPFAR. MACRO operates across multiple projects including the IMPACT Project (KP Project), with activities spanning Lilongwe, Blantyre, and Mzuzu.




1.2  Purpose of this Document

This System Design Document (v4.0) is the optimised, production-hardened evolution of v3.0. It preserves the full functional specification of v3.0 and introduces twelve architectural optimisations that address production readiness gaps: event-driven side-effect handling, zero-trust security posture, full-stack observability, feature flag governance, an outbound integration layer for donor systems, a formal data archival and retention policy, formalised cache invalidation, blue/green deployment, circuit breaker resilience, database partitioning for high-volume tables, API contract testing, and tenant isolation readiness.




1.3  Scope

All functional scope from v3.0 is retained without modification. The following areas are added or materially changed:


Event-driven architecture replacing synchronous side effects in the workflow engine

3. System Overview


3.1  System Vision

MACRO PMS will be a secure, web-based platform that digitizes all current paper forms and enforces the organisation's approval hierarchies, compliance rules, and audit requirements electronically. Every document type currently produced on paper will have a corresponding digital form in the system. The system will maintain a complete, immutable audit trail of all approvals, modifications, and financial transactions, and will automatically manage budget balances in real time.

4. Functional Modules

Modules 1-13 are defined in v3.0 and carried forward. This section documents only the material changes to existing modules and the four new modules (4.14-4.17).




4.1  Module 1: Security, Authorization, and Infrastructure (Optimised) [CHANGED]

All v3.0 security controls are retained. The following optimisations extend the module to a zero-trust posture.




4.1.7  Zero-Trust Inter-Service Security [NEW]

v3.0 relied on network-level isolation (private subnets) as the primary trust boundary between backend components. v4.0 removes implicit trust of network boundaries and enforces identity verification on every inter-service call:


All backend-to-backend HTTP calls (API server to worker, API server to PDF generator) use mutual TLS (mTLS). Each service has its own TLS certificate issued by a private CA managed in AWS Private Certificate Authority or HashiCorp Vault.


Every inter-service request carries a signed Authorization header using HMAC-SHA256 over the request body and timestamp. The receiving service verifies the signature against a shared secret rotated every 90 days.


Service-to-database connections are authenticated using individual PostgreSQL roles per service (api_service, worker_service, migration_service). Each role has only the minimum permissions required for its function. The api_service role cannot execute DDL statements; the migration_service role cannot read financial data at rest.


Redis connections use ACL-based authentication (Redis 7 ACLs). The API service can read and write session keys but cannot FLUSHALL. The worker service can manage job queues but cannot read session tokens.




4.1.8  Field-Level Access Control [NEW]

Certain data fields require stricter access than the entity-level RBAC provides. v4.0 introduces a field-level access control layer that masks or redacts sensitive fields based on the requesting user's role:

4.1.9  API Contract and Versioning [NEW]

The REST API is defined by an OpenAPI 3.1 specification file maintained in the project repository as the single source of truth. All request and response schemas, path parameters, query parameters, and error codes are defined in the spec. Contract tests are auto-generated from the spec and run in CI on every pull request. Any endpoint that fails its contract test blocks merge.

Breaking changes (removal of a field, change of field type, removal of an endpoint) require a new API version namespace (/api/v2/). The previous version continues to be served for a minimum of six months after v2 goes live, with a deprecation header on all v1 responses.





4.3  Module 3: Review and Approval Workflow Engine (Optimised) [CHANGED]

The FSM transition function, state definitions, and guard conditions from v3.0 are unchanged. The material change in v4.0 is the decoupling of side effects from the transition function into an asynchronous, event-driven architecture.




4.3.4  Event-Driven Side Effects [CHANGED]

In v3.0, side effects (budget posting, voucher generation, notification delivery) executed synchronously within the same database transaction as the state transition. This created tight coupling: a failure in email delivery could roll back a valid budget commitment, and adding a new side effect required modifying the core workflow engine.

v4.0 decouples side effects using a transactional outbox pattern:




The workflow engine's attemptTransition function performs only two operations within its database transaction: (a) update the document state, and (b) insert a domain event record into the outbox table (event_type, payload, created_at, processed_at = NULL).


A background poller (running on a configurable interval, default 500ms) reads unprocessed events from the outbox, dispatches them to registered subscribers, and marks them as processed.


Each subscriber handles exactly one concern: BudgetSubscriber posts commitments and disbursements, VoucherSubscriber generates payment vouchers, NotificationSubscriber sends emails and push notifications, IntegrationSubscriber forwards events to the webhook layer (Module 4.16).


Subscriber failures do not affect the transition or other subscribers. A failed subscriber's event is retried up to five times with exponential backoff (1s, 2s, 4s, 8s, 16s). After five failures, the event is moved to a dead-letter queue and an alert is sent to the System Administrator.


The outbox table provides a complete, ordered log of all domain events. This enables event replay for debugging, re-processing after a bug fix, and building new read models or integrations without touching historical data.

4.14  Module 14: Observability [NEW]

This module replaces the ad-hoc monitoring described in v3.0 NFR Section 7.5 with a structured, three-pillar observability stack: logs, traces, and metrics. All three pillars share a common correlation ID that links a user action to every log line, trace span, and metric data point it produces.




4.14.1  Structured Logging

All application logs are emitted in structured JSON format with the following mandatory fields: timestamp (ISO 8601 with timezone), level (DEBUG/INFO/WARN/ERROR), correlation_id (UUID injected by API gateway middleware on every inbound request), service (api, worker, scheduler), message, and context (JSONB with endpoint, user_id, role, tenant_id). Log levels are configurable per environment via environment variable without restart. Sensitive data (passwords, tokens, bank account numbers) is never logged — a static analysis rule in CI rejects any log statement referencing a field on the sensitive-fields list.




4.14.2  Distributed Tracing

Distributed tracing is implemented using OpenTelemetry SDK instrumentation. Every inbound API request starts a root trace span. The span propagates through all internal calls (database queries, Redis operations, S3 uploads, event bus dispatches, and worker job executions) as child spans. Trace context is propagated via the W3C Trace Context header (traceparent).

Key spans are annotated with domain-specific attributes: workflow transitions carry document_type and from_state/to_state; budget operations carry budget_line_id and transaction_amount; search operations carry query_text and result_count. Traces are exported to Jaeger (self-hosted) or AWS X-Ray and retained for 30 days.




4.14.3  Metrics and SLI/SLO Definitions

The following Service Level Indicators (SLIs) are defined, each with a corresponding Service Level Objective (SLO):

SLO burn-rate alerts are configured: if the error budget for the 30-day SLO is consumed at a rate that would exhaust it within 1 hour (fast burn) or within 3 days (slow burn), an alert fires to the on-call engineer. Burn-rate alerts replace simple threshold alerts because they account for both severity and duration.




4.14.4  Alerting and Runbooks

Every alert is linked to a runbook document in the project wiki. The runbook contains: alert description, likely root causes, diagnostic steps (including specific log queries and trace searches), remediation actions, and escalation path. Alerts without a linked runbook are not permitted — the CI pipeline rejects alert rule changes that do not include a runbook URL.





4.15  Module 15: Feature Flags [NEW]

All new functionality in MACRO PMS is shipped behind feature flags. A feature flag is a runtime-configurable boolean or percentage gate that controls whether a specific feature is active for a given user, role, project, or percentage of traffic.




4.15.1  Flag Types

Flag Type

Behaviour

Boolean

On or off globally. Used for kill switches and completed rollouts awaiting cleanup.

Role-based

Active for specified roles only. Used during phased rollout: enable for System Administrator first, then Finance, then all staff.

Project-based

Active for specified projects only. Used to pilot a feature on the IMPACT project before rolling out to all projects.

Percentage

Active for N% of users, determined by a hash of user ID (stable — same user always gets the same result). Used for gradual rollout with monitoring between increments.




4.15.2  Flag Lifecycle

Every flag has a lifecycle: CREATED (defined in code, defaults to off), ROLLING_OUT (actively being enabled for segments), FULLY_ROLLED_OUT (on for all users), and RETIRED (code path for the old behaviour removed). Flags that remain in FULLY_ROLLED_OUT state for more than 30 days trigger a cleanup reminder in the weekly engineering report. Flag state is stored in the database and cached in Redis with a 60-second TTL. Flag evaluation is synchronous and adds less than 1ms to request processing time.




4.15.3  Kill Switch

Any flag can be instantly set to off via an admin endpoint or the System Administrator UI. This immediately disables the feature for all users without a deployment. Kill switch activations are logged in the audit trail and trigger a notification to the engineering team. This is the primary rollback mechanism for features that cause production issues — faster and safer than redeploying the previous code version.





4.16  Module 16: Integration and Webhook Layer [NEW]

This module provides a structured mechanism for MACRO PMS to push data to external systems, starting with the PEPFAR DATIM reporting platform. It is designed as a generic webhook dispatcher that can be extended to additional integrations without modifying core application logic.




4.16.1  Webhook Configuration

Each integration is configured as a webhook subscription stored in the database:

Field

Description

subscription_id

UUID primary key.

name

Human-readable label (e.g. PEPFAR DATIM Activity Sync).

target_url

HTTPS endpoint to deliver payloads to.

event_types

Array of domain event types this subscription listens to (e.g. [advance.approved, voucher.paid]).

signing_secret

HMAC-SHA256 secret used to sign payloads. Stored encrypted in AWS Secrets Manager.

transform_template

JSONata or Handlebars template that transforms the domain event payload into the format required by the target system.

active

Boolean. Inactive subscriptions are skipped by the dispatcher.

retry_policy

Max retries (default 5), backoff base (default 2s), backoff multiplier (default 2x).




4.16.2  Delivery Guarantees

The webhook dispatcher is an event bus subscriber (IntegrationSubscriber). When it receives a domain event, it: (1) evaluates which subscriptions match the event type, (2) applies the transform template to produce the outbound payload, (3) computes an HMAC-SHA256 signature over the payload using the subscription's signing secret and includes it in the X-Webhook-Signature header, (4) delivers the payload via HTTPS POST, (5) expects a 2xx response within 10 seconds.

On failure (network error, non-2xx response, timeout), the delivery is retried with exponential backoff up to the configured maximum. After all retries are exhausted, the event is moved to a dead-letter queue. The System Administrator can view the dead-letter queue, inspect payloads, and manually retry or discard failed deliveries. A daily digest alert lists any events in the dead-letter queue.





4.17  Module 17: Data Archival and Retention [NEW]

This module defines a formal data lifecycle policy that balances operational performance with long-term compliance requirements (PEPFAR requires seven-year record retention for grant-funded transactions).




4.17.1  Data Tiers

Tier

Retention

Current + 1 fiscal year

4.17.2  Archival Process

Tier promotion runs as a scheduled job on the first day of each fiscal year. The job identifies all records in the hot tier whose associated project's grant period ended more than one fiscal year ago. These records are copied to the warm tier (read-only tablespace), verified by row-count and checksum comparison, and then removed from the hot tier's active partitions. The same process promotes warm-tier data to cold storage after the 3-year threshold.

Archival operations are idempotent — running the job twice produces no duplicate data. Each archival run is logged with record counts, source and destination checksums, and execution duration. Failures halt the job and alert the System Administrator; no partial archives are committed.




4.17.3  Impact on Application Queries

Application queries default to the hot tier. Users can opt in to searching warm-tier data via a Historical data toggle on search and filter panels. This toggle routes the query to the read replica's warm tablespace. Cold-tier data is not queryable inline — it must be restored to the warm tier first. The data tier of each record is displayed as a subtle badge (e.g. Archived - 2024) on record detail views.

9. Infrastructure Architecture


9.1  Network Topology

Network topology is unchanged from v3.0 (public subnet for ALB/WAF/NAT, private subnet for all application and data components). The addition of mTLS (Section 4.1.7) means that even within the private subnet, all inter-component communication is encrypted and mutually authenticated.




9.2  Component Topology

Component topology and sizing are unchanged from v3.0. One addition: the Prometheus and Jaeger containers are deployed on the worker instance or a dedicated monitoring instance (t3.small), separate from the API servers, to avoid resource contention during high observability load.




9.3  Deployment Strategy: Blue/Green [CHANGED]

v3.0 used a rolling update deployment strategy. v4.0 upgrades to blue/green deployment for safer, faster rollbacks:




Two identical production environments (Blue and Green) are maintained behind the Application Load Balancer. At any time, one is live (serving traffic) and the other is idle (standby).


On deployment, the new Docker image is deployed to the idle environment. Smoke tests run against the idle environment using a dedicated internal URL — not live traffic.


Database migrations run before the traffic switch. All migrations must be backward-compatible (additive columns, new tables) so that the live environment continues to operate against the migrated schema. Destructive migrations (column drops, renames) are deferred to a cleanup migration in a subsequent release after the old code path is no longer in use.


Once smoke tests pass and the System Administrator gives manual approval, the ALB target group is switched from the live environment to the idle environment. Traffic cutover is instant (single ALB target group swap, typically under 5 seconds).


The previous live environment is retained as a rollback target for 24 hours. If a production issue is detected, the ALB target group is switched back to the previous environment — rollback time is under 30 seconds.


After the 24-hour rollback window, the old environment is updated to match the new release, becoming the new standby for the next deployment.



This replaces the rolling update strategy from v3.0. The key advantages are: instant rollback (ALB swap vs. redeployment), full pre-production smoke testing against real infrastructure (not just health checks), and elimination of the mixed-version window that rolling updates create.




9.4  Disaster Recovery [CHANGED]

WAL-based continuous archiving to S3 replaces nightly pg_dump as the primary backup mechanism. This reduces RPO from 24 hours to under 1 hour (bounded by WAL shipping lag, typically under 30 seconds). Point-in-time recovery is available to any second within the WAL retention window (default 7 days).

Weekly full pg_dump snapshots are retained as a secondary backup for cold-storage archival. Monthly automated restore drills (upgraded from quarterly) verify both WAL-based recovery and pg_dump recovery paths. Document store backups via S3 versioning and cross-region replication are unchanged from v3.0.

11. Risks and Mitigations

All v3.0 risks are retained. The following additional risks are introduced or re-assessed by v4.0 optimisations:



Risk

Likelihood

Mitigation

Event bus message loss

Low

[NEW] Transactional outbox pattern guarantees at-least-once delivery. Events are committed in the same DB transaction as the state change. Background poller retries with exponential backoff. Dead-letter queue for investigation.

Event subscriber ordering issues

Low

[NEW] Outbox events are processed in creation order (BIGSERIAL PK). Each subscriber processes events sequentially. No cross-subscriber ordering dependency.

Feature flag misconfiguration

Medium

[NEW] All flag changes are audit-logged. Kill switch provides instant rollback. Flag changes require System Administrator role. Percentage rollouts use stable hashing to prevent user experience oscillation.

Webhook delivery failure to DATIM

Medium

[NEW] Exponential backoff retries (up to 5). Dead-letter queue with daily digest alert. Manual retry capability. Circuit breaker prevents cascading failures if DATIM is down for extended periods.

Observability data volume exceeds storage

Low

[NEW] Trace sampling at 10% for non-error requests in production (100% in staging). Log retention: 90 days hot, then cold storage. Metric retention: 15-second resolution for 24 hours, downsampled to 1-minute for 30 days.

Blue/green deployment with incompatible DB migration

Low

[NEW] All migrations must be backward-compatible (expand-contract pattern). Destructive changes deferred to cleanup migration. Migration CI step validates backward compatibility by running the migration and then booting the previous application version against the migrated schema.

mTLS certificate expiry causes service outage

Low

[NEW] 90-day certificates with auto-rotation at 60 days. Monitoring alert fires at 14 days before expiry. Rotation runs as a scheduled job with failure alerting.

Data archival corrupts or loses records

Low

[NEW] Archival is idempotent and transactional. Row-count and checksum verification before deletion from source tier. Archival job halts on any mismatch. Finance Manager sign-off required for permanent deletion at 7-year mark.

Appendix F: Glossary Additions (v4.0) [NEW]

Term

Definition

Blue/Green Deployment

A deployment strategy that maintains two identical production environments and switches traffic between them for zero-downtime releases and instant rollback.

Circuit Breaker

A resilience pattern that detects failures to an external dependency and stops making requests until the dependency recovers, preventing cascade failures.

Correlation ID

A UUID assigned to every inbound API request and propagated through all internal processing, linking logs, traces, and metrics to a single user action.

DATIM

Data for Accountability, Transparency, and Impact Monitoring — PEPFAR's reporting platform.

Dead-Letter Queue

A holding area for messages or events that could not be processed after exhausting all retry attempts. Requires manual intervention.

Domain Event

A record of something that happened in the system (e.g. advance.approved). Stored in the outbox table and dispatched to subscribers asynchronously.

Feature Flag

A runtime-configurable gate that controls whether a specific feature is active, enabling phased rollout and instant rollback without redeployment.

mTLS

Mutual TLS — a TLS connection where both client and server present and verify certificates, ensuring both parties are authenticated.

OpenAPI

A specification format for describing REST APIs. Used as the single source of truth for the MACRO PMS API contract.

OpenTelemetry

An open-source observability framework for generating, collecting, and exporting traces, metrics, and logs.

Outbox Pattern

A pattern where domain events are written to a database table within the same transaction as the state change, then dispatched asynchronously by a background process.

SLI

Service Level Indicator — a quantitative measure of system behaviour (e.g. request latency, availability).

SLO

Service Level Objective — a target value for an SLI over a defined period (e.g. 99.5% availability over 30 days).

WAL

Write-Ahead Log — PostgreSQL's transaction log. Continuous archiving of WAL enables point-in-time recovery.
